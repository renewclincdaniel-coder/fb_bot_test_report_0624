# 智能客服現況 × 對話終點分析 對照報告

更新日期：2026-06-24

本報告把兩件已經各自完成的事實放在一起對照：

1. **對話終點分析**：8,637 段歷史對話，每段最後「客人去了哪裡」的分類結果。
2. **Facebook_CS 智能客服系統現況**：目前實際已實作的模組、機制，以及 DeepEval 測試分數。

目的：用前因後果說清楚——**目前智能客服「能 cover」哪些流程/情境（為什麼能）、「接不住」哪些情境（缺什麼），以及一個關鍵事實：佔比最大的「無回覆流失」受 Meta 平台規則限制，智能客服（bot）無法主動發訊息撈回（詳見第五章）**。

> 本報告所有數字皆來自既有檔案，未做任何推估或編造。資料來源見文末附錄。

## 一、先看事實：8,637 段對話最後去了哪裡

把終點分類的分母改成全部 8,637 段（含廣告回覆、直接私訊、貼文留言、貼文回覆四個入口），結果如下：

| 終點大類 | 對話數 | 佔全體 8,637 | 白話解讀 |
| --- | ---: | ---: | --- |
| **無回覆 / 流失** | **6,934** | **80.28%** | 對話斷掉，客人沒有再回 |
| 明確轉換承接 | 1,365 | 15.80% | 留了聯絡方式或進到預約協調 |
| 客戶待客服回覆 | 226 | 2.62% | 最後一句在客人這邊，客服可再接 |
| 婉拒 / 預算疑慮 | 112 | 1.30% | 客人明確表達不考慮 |

**結論一句話：每 5 段對話就有 4 段是「斷在半路、沒有回覆」。這就是智能客服最大的戰場。**

再把「無回覆/流失」拆細，看斷點到底發生在哪一步（全體 8,637 為分母）：

| 流失細項 | 對話數 | 佔全體 | 斷在哪一步 |
| --- | ---: | ---: | --- |
| 快速回覆後無後續 | 2,091 | 24.21% | 客人點了一次按鈕就沒下文 |
| 客服開場或釐清後無回覆 | 2,067 | 23.93% | 客服打完招呼/問完需求，客人不回 |
| 其他客服回覆後無回覆 | 848 | 9.82% | 停在一般客服訊息 |
| 客服提供預約或轉私域後無回覆 | 818 | 9.47% | 客服丟了門診/LINE，客人沒接 |
| 客服要求聯絡方式後無回覆 | 680 | 7.87% | 客服要 LINE/電話，客人不給 |

> 補充：貼文留言入口中有 240 段是「私訊串內根本沒有客戶訊息」（系統把留言事件當入口，但私訊沒有客人後續），這類不是真流失，是資料口徑造成的，解讀時要分開看。

### 補充：對話性質分布 —— 客訴（高風險）只佔 0.30%

換一個角度，把同一批 8,637 段依「對話性質」分類（來源 `Cleaned_Chat_Dataset_V2`，三類加總 8,501 + 110 + 26 = 8,637，與總數完全吻合）：

| 對話性質 | 對話數 | 佔全體 8,637 |
| --- | ---: | ---: |
| 一般諮詢（general_inquiry） | 8,501 | 98.43% |
| 正面／成交傾向（positive_sales） | 110 | 1.27% |
| **負面／客訴（negative_complaints）** | **26** | **0.30%** |

**重點：客訴（高風險）只佔 0.30%（26 段）**，數量極少。但因為涉及醫療、法律與品牌風險，是智能客服「**寧可不答、也不能答錯**」的最高優先；設計上一律走「安撫 → 收必要資訊 → 立即轉真人」，不交給模型自由判斷（處理範例見第三章）。

> 註：`negative_complaints` 是「負面情緒／客訴」的廣義分類，含預算抱怨、不滿語氣等；其中真正高風險的醫療／法律案例（毀容、賠償、術後異常）是更小的子集。

## 二、為什麼會流失：終點對應到對話的哪一步

把終點和對話流程對齊，可以看到流失集中在「漏斗最前段」與「轉換臨門一腳」兩處：

```
客人進來 → 點快速回覆 → 客服開場/釐清 → 療程/問價說明 → 要求聯絡/提供預約 → 留聯絡/預約
   │            │              │                │                  │              │
   │       2,091 斷這  2,067 斷這         （承接中）      680+818 斷這      1,365 成功
   │       (24.21%)   (23.93%)                          (16% 臨門流失)    (15.80%)
```

- **最前段流失最大**：快速回覆後無後續（24.21%）＋ 開場釐清後無回覆（23.93%），合計約 **48%**，幾乎一半的對話在「還沒進入實質諮詢」就斷了。
- **臨門一腳也在漏**：要求聯絡方式後無回覆（7.87%）＋ 提供預約/轉私域後無回覆（9.47%），合計約 **17%**，這些客人已經被推到轉換點，卻沒跨過去。

這兩個結論直接決定了智能客服「該優先補什麼」。

## 三、智能客服目前「能接住」什麼（已實作 + 為什麼）

以下都是 Facebook_CS 系統**已經實作**的模組與機制（來源：系統架構知識庫、優化報告 Phase 進度），對應它能覆蓋的終點情境。

### 3.0 架構簡單說明（一句話看懂流程）

```
客人發訊息
  → ① FastAPI 收訊：驗簽、去重、偵測「真人小編是否已介入」（介入則 AI 暫停）
  → ② Celery 排隊處理（避免 FB 20 秒逾時），先擋掉惡意注入
  → ③ FacebookAgent 判斷「情緒 + 意圖」，分流到對的處理：
        • 高憤怒 / 客訴 / 術後 → 安撫 → 收必要資訊 → 轉真人（不自由發揮）
        • 隱私查詢（查預約/點數）→ 不直接查，引導留姓名電話轉專人
        • 一般諮詢（療程/價格/衛教）→ RAG 查知識庫 → 生成回答
  → ④ 繁體中文強制 + 智能追問（用 ||| 拆兩則，補一句業務關懷）
  → 回覆客人
```

- **知識庫 RAG** = ChromaDB 向量檢索 + BM25 關鍵字 + Reranker 重排，確保回答有依據、不亂編。
- **安全守門最優先**：客訴/術後/法律風險不靠模型自由判斷，固定走安撫＋轉真人。
- **真人可隨時接手**：小編一回覆，AI 自動暫停 45 分鐘，不搶話。

### 3.1 已實作的核心模組

| 模組 / 機制 | 技術實作 | 能接住的情境 | 對應終點 |
| --- | --- | --- | --- |
| 入口接收與去重 | FastAPI Webhook、HMAC 驗簽、Redis 去重、Echo 偵測 | 廣告回覆 / 私訊 / 留言進線 | 全部入口識別 |
| 智能路由 | `FacebookAgent` 依情感+意圖分流 | 一般諮詢 vs 負面情緒 vs 隱私 vs 客訴 | 療程探索、問價 |
| RAG 知識回答 | ChromaDB + BM25 + BAAI Reranker + OpenRouter | 療程介紹、恢復期、價格範圍 | 療程/困擾探索、純價格評估 |
| **即時智能追問** | 回答後用 `\|\|\|` 拆兩則，延遲 1.5s 補一句業務關懷 | 回答後主動再問一句，降低「答完就斷」 | **快速回覆後無後續（部分）** |
| 新舊客問候 | 無 `user_session` 才加問候；舊客直接答 | 開場不重複、不機械 | 開場或釐清後無回覆（部分） |
| Lazy profile 蒐集 | 對話中順勢收姓名/年齡/病史，不用表單 | 降低留資料的摩擦 | 要求聯絡方式後無回覆（部分） |
| 情緒安撫循環 | `anger_score>7` 進安撫；3 次失敗轉真人 | 負面情緒、客訴 | 婉拒/客訴 |
| 安全 / 隱私守門 | Prompt injection 過濾、隱私查詢不直答、術後客訴分流轉真人 | 術後異常、賠償、個資查詢 | 高風險 handoff |
| 真人介入暫停 | admin echo → `pause_bot 45min`，不搶話 | 真人接手時 AI 自動讓位 | handoff |

### 3.2 情境處理範例（智能客服怎麼接不同情境）

以下用三個主要情境，說明系統實際怎麼分流與回應（話術依現有人設與機制；說明性範例）。

**① 衛教／療程探索 —— RAG 回答 + 智能追問**
```
客人：想了解皮秒雷射
AI：皮秒雷射主要針對毛孔粗大與斑點…效果顯著。|||請問您之前有做過雷射相關保養嗎？😊
```
→ 先用知識庫 RAG 給有依據的衛教，再用 `|||` 拆兩則、延遲 1.5 秒補一句關懷，把球丟回客人；必要時分享 YouTube 衛教影片。這是對話量最大的主流程（廣告回覆 83% 屬療程探索）。

**② 預約 —— Booking 流程收集欄位，插問先答再回**
```
客人：我想約這週五看診
AI：好的～想先幫您安排！方便提供「姓名 / 電話 / 想了解的療程 / 方便的時段」嗎？
客人：那三層次拉皮多少錢？
AI：（先回價格）三層次拉皮優惠價約 36 萬，依臉況不同…|||那我這邊先幫您把週五時段保留，您的大名與電話是？
```
→ Booking handler 負責收姓名/電話/療程/時段；中途被問價格時，先用 RAG 回答，再「回到」原本缺的欄位，最後產生交接摘要給專員。

**③ 客訴／高風險 —— 安撫 → 不自由判斷 → 立即轉真人**
```
客人：你們把我臉做壞了，我要求賠償！
AI：非常抱歉讓您有不愉快的體驗，我們非常重視您的感受😔 我立即為您安排專責人員協助處理。
（系統：偵測高風險 → 安全守門過濾「手術失敗/醫師疏失」等禁語 → 不做醫療或法律判斷 → 轉真人主管）
```
→ 這正是第一章 0.30% 客訴的處理方式：情緒安撫最多嘗試 1–2 次，**一律轉真人**，不讓模型自由回應醫療/法律問題。

**④ 隱私查詢 —— 不直接查，引導轉專人**
```
客人：幫我查我的預約紀錄
AI：為保護您的隱私，無法直接查詢，請留「姓名+電話」請專人為您確認唷！👩‍⚕️
```

### 3.3 測試結果：能跑，但品質尚未過關（事實陳述）

DeepEval 2026-05-26 scored baseline（25 cases）與 Phase 4 最新進度：

| 能力 | 測試分數 | 判讀 |
| --- | --- | --- |
| RAG 不亂編（Faithfulness） | 0.944 → Phase 4C **1.000** | ✅ 回答有根據、不捏造 |
| 答題相關性（Answer Relevancy） | 0.983 | ✅ 答得到點上 |
| 檢索精準度（Retrieval Relevancy） | 0.771 → 0.743 | ⚠️ 知識庫 evidence 不夠，價格類尤其弱 |
| 預約流程完整度（Completeness） | 0.59 → Phase 4 **0.717** | ❌ 未達 0.80 門檻 |
| 多輪上下文 | 0.66 → Phase 4B **0.84** | ⚠️ 接近通過，仍未穩 |
| 繁體中文品質 | **25/25 全 fail**（0.648） | ❌ 內部 `[SPLIT:*]` marker 漏進輸出 |

**重點：系統「跑得動、答得出、不亂編」已經做到；但「預約收尾、多輪記憶、輸出乾淨度」這三項還沒過 gate，整體 gate 目前 = failed。**

## 四、智能客服目前「接不住」什麼（缺口 + 量化）

把終點分析的最大流失，對照系統現況，要分成兩種性質：**(A) 可開發補強的缺口**、**(B) 平台規則限制（無法靠開發解決）**。

| 最大流失終點 | 量化 | 系統現況 | 性質 |
| --- | ---: | --- | --- |
| 快速回覆後無後續 | 2,091（24.21%） | 同一輪有即時追問（`\|\|\|`）；但客人一旦離線就斷 | **(B) 平台限制**：跨日再接觸需主動發訊息，bot 不可為（見五） |
| 開場/釐清後無回覆 | 2,067（23.93%） | 有問候策略，但開場後不回就斷 | **(B) 平台限制**：同上，無法隔日自動再問 |
| 提供預約/轉私域後無回覆 | 818（9.47%） | 有 booking flow，但 completeness 0.717 未達標 | **(A) 可補**：預約收尾不穩（對話視窗內可優化） |
| 要求聯絡方式後無回覆 | 680（7.87%） | 有 lazy profile | **(A) 可補**：對話視窗內可加替代路徑話術 |

**核心事實一句話：智能客服（bot）只能在「客人最後互動的 24 小時對話視窗內」運作。一旦客人已讀不回、隔天才想起來，受 Meta 平台規則限制，bot 無法主動發訊息把他撈回來——這不是開發缺口，是平台限制（詳見第五章）。而「對話已中斷」正是 80.28% 流失的主體。**

## 五、關鍵事實：智能客服（Bot）無法主動發訊息

### 5.1 事實陳述（平台規則，非開發缺口）

經查證 Meta Messenger 2026 年最新平台規則，**智能客服（bot）沒有合規的「自動主動發訊息」管道**：

| 管道 | bot 能否自動用 | 2026 現況 |
| --- | --- | --- |
| 24 小時對話視窗內自由訊息 | ✅ 可 | 唯一能自動發的範圍 |
| 標準訊息標籤（CONFIRMED_EVENT / ACCOUNT / POST_PURCHASE） | — | ❌ 2026 陸續停用 |
| 一次性通知 OTN | — | ❌ 已淘汰 |
| 行銷訊息 Marketing Messages（原 Recurring Notifications） | — | ❌ 僅 EU/UK/日/韓/澳，**台灣不適用** |
| 付費贊助訊息 Sponsored Messages | （需付費、屬廣告投放） | 與智能客服 bot 自動化無關 |
| 真人客服 7 天內手動回覆（HUMAN_AGENT） | ❌ 限真人 | 屬人力作業，非 bot 能力 |

→ **結論：超過「客人最後互動 24 小時」之後，智能客服（bot）在現行平台規則與台灣地區條件下，無法自動主動發訊息。** 因此 80.28% 流失中「對話已中斷、需要主動再接觸」的部分，**不是靠開發就能解決，是平台限制**。

### 5.2 因此，智能客服的有效範圍 ＝「客人有回應的當下對話」

- bot 的價值集中在 **「客人還在線上、24 小時對話視窗內」** 把對話接好、往成交推進。
- 直白說：智能客服**無法「把已流失的人撈回來」**，但可以**「讓還在對話中的人不要流失、並推進到留聯絡/預約」**。
- 所以優化重點不是「設計隔日追蹤」（做不到），而是**在客人還在的當下，把該答的答好、該收的收好**。

### 5.3 目前「可以 Cover」的流程／情境總覽（對話視窗內）

以下是智能客服**現在就能處理**的流程，並對應到它能觸及的對話量：

| 可 cover 的流程／情境 | 對應對話量（事實） | 智能客服現在能做什麼 | 狀態 |
| --- | --- | --- | --- |
| 療程／困擾探索 | 廣告回覆 83.34%（5,978 段）主任務 | 辨識療程與部位、釐清困擾、簡短衛教、引導現場評估 | ✅ 可 cover（檢索證據待補） |
| 問價／價格評估 | 問價節點 2,381 段 | RAG 回答價格範圍與計價邏輯，再導向評估 | ⚠️ 可 cover，價格 evidence 待補強 |
| 同一輪即時追問 | 快速回覆／答題後 | `\|\|\|` 拆兩則，補一句業務關懷把球丟回客人 | ✅ 可 cover |
| 多輪上下文 | 連續問價／問痛感 | 用前文療程回答後續問題，不重問 | ⚠️ 0.84 接近通過 |
| 聯絡／轉私域 | 客服要求聯絡 2,500 段 | 順勢收姓名/電話/LINE，產生 handoff 摘要 | ✅ 可 cover |
| 預約資料收集 | 預約協調相關 | 收集缺漏欄位，插問先答再回預約 | ⚠️ 0.717 收尾未達標 |
| 安全／客訴／術後 | 高風險情境 | 安撫、收集必要資訊、立即轉真人 | ✅ 可 cover |
| 真人介入／Echo | — | 真人接手時 AI 自動暫停，不搶話 | ✅ 可 cover |
| 隱私查詢 | — | 不直接查詢，引導留資料轉專人 | ✅ 可 cover |

### 5.4 「無法 Cover」的部分（明確標示）

| 無法 cover 的情境 | 原因 |
| --- | --- |
| 對話已中斷、隔日／數日後主動再接觸（撈回流失） | ❌ 平台限制：bot 24h 外無合規自動發送管道（見 5.1） |
| 任何超過 24 小時的 bot 主動訊息 | ❌ 同上 |

> 若未來要處理「跨日再接觸」，只能走**真人客服手動回覆（7 天內）**的人力路徑，屬營運/人力範疇，不在智能客服（bot）能力範圍，本報告不納入 bot 規劃。

## 六、其他需補強項（依優先序，皆有測試佐證）

| 優先 | 項目 | 現況事實 | 為什麼要補 |
| --- | --- | --- | --- |
| P0 | 安全 / handoff hard fail = 0 | medical_safety 仍有 1 筆 0.70 | 醫療法律風險，錯答比不答更危險 |
| P1 | 繁中輸出乾淨度 | 繁中品質 25/25 fail，`[SPLIT:*]` 漏出 | 內部 marker 出現在客人眼前，直接傷品牌 |
| P2 | 預約流程收尾 | Completeness 0.717，未達 0.80 | 對應「提供預約後無回覆 818 段」 |
| P3 | RAG 知識庫 evidence | Retrieval Relevancy 0.743 | 對應問價情境，價格類證據最弱 |

### 7.4 範圍說明：不含「跨日主動追蹤」

原先設想的「跨日主動追蹤」已確認受 **Meta 平台規則限制**——智能客服（bot）在客人最後互動 24 小時後，無合規的自動主動發訊息管道（見第五章 5.1）。因此本時程**僅聚焦「對話視窗內」的品質與轉換優化**，不包含任何 bot 主動發訊息的開發。

### 7.5 假設與風險

- **假設**：工程人力連續投入、不做大規模架構重寫（orchestrator 拆分可漸進完成）、價格 KB 內容可即時提供。
- **風險**：
  1. 繁中品質若不只是 marker 洩漏、還涉及語氣與格式門檻，可能多花 3–5 天。
  2. RAG evidence 補齊依賴內容整理速度，是時程最大不確定來源之一。
  3. Live A/B 需足夠進線量才有統計顯著性，驗證時間視流量而定。

## 八、總結：一張圖看懂現況

```
                  對話視窗內（bot 有效範圍）       平台限制 / 待補
                  ──────────────────────       ──────────────
入口/路由     ✅ Webhook、去重、情感路由
療程探索      ✅ RAG（不亂編 Faithfulness 1.0）   ⚠️ 檢索證據不足 0.74（可補）
問價          ✅ 能答                            ⚠️ 價格 evidence 弱（可補）
同輪追問      ✅ |||即時拆兩則
開場          ✅ 新舊客問候
預約          ⚠️ 有流程                          ❌ 收尾完整度 0.717 未達標（可補）
聯絡方式      ✅ lazy profile                    ⚠️ 對話內可加替代話術（可補）
安全/客訴     ✅ 守門+轉真人                      ⚠️ 1 筆 medical safety 未達標（可補）
輸出品質      —                                 ❌ 繁中 25/25 fail（可補）
跨日撈回流失   —                                 ⛔ 平台限制：bot 24h 外不能主動發訊息
```

**一句話總結：智能客服目前「在客人還在對話的當下，能正確回答、推進成交」已有一定水準；剩下缺口（預約收尾、繁中輸出、價格證據）都在「對話視窗內」、皆可開發補強。但佔 80.28% 流失的主體是「對話已中斷」，受 Meta 平台規則限制，智能客服（bot）無法主動發訊息撈回——這是平台事實，非開發項。因此下一步重點明確：把「對話視窗內」的品質修到能上線，讓還在線上的客人不流失。**

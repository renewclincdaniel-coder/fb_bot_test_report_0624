# 智能客服現況 × 對話終點分析 對照報告

更新日期：2026-06-24

本報告把兩件已經各自完成的事實放在一起對照：

1. **對話終點分析**：8,637 段歷史對話，每段最後「客人去了哪裡」的分類結果。
2. **Facebook_CS 智能客服系統現況**：目前實際已實作的模組、機制，以及 DeepEval 測試分數。

目的：用前因後果說清楚——**目前智能客服「能接住」哪些情境（為什麼能）、「還接不住」哪些情境（缺什麼），以及佔比最大的「無回覆流失」該怎麼補**。

> 本報告所有數字皆來自既有檔案，未做任何推估或編造。資料來源見文末附錄。

## 一、先看事實：8,637 段對話最後去了哪裡

把終點分類的分母改成全部 8,637 段（含廣告回覆、直接私訊、貼文留言、貼文回覆四個入口），結果如下：

| 終點大類 | 對話數 | 佔全體 8,637 | 白話解讀 |
| --- | ---: | ---: | --- |
| **無回覆 / 流失** | **6,934** | **80.28%** | 對話斷掉，客人沒有再回 |
| 明確轉換承接 | 1,365 | 15.80% | 留了聯絡方式或進到預約協調 |
| 客戶待客服回覆 | 226 | 2.62% | 最後一句在客人這邊，客服可再接 |
| 婉拒 / 預算疑慮 | 112 | 1.30% | 客人明確表達不考慮 |

**結論一句話：每 5 段對話就有 4 段是「斷在半路、沒有回覆」。這就是智能客服最大的戰場。**

再把「無回覆/流失」拆細，看斷點到底發生在哪一步（全體 8,637 為分母）：

| 流失細項 | 對話數 | 佔全體 | 斷在哪一步 |
| --- | ---: | ---: | --- |
| 快速回覆後無後續 | 2,091 | 24.21% | 客人點了一次按鈕就沒下文 |
| 客服開場或釐清後無回覆 | 2,067 | 23.93% | 客服打完招呼/問完需求，客人不回 |
| 其他客服回覆後無回覆 | 848 | 9.82% | 停在一般客服訊息 |
| 客服提供預約或轉私域後無回覆 | 818 | 9.47% | 客服丟了門診/LINE，客人沒接 |
| 客服要求聯絡方式後無回覆 | 680 | 7.87% | 客服要 LINE/電話，客人不給 |

> 補充：貼文留言入口中有 240 段是「私訊串內根本沒有客戶訊息」（系統把留言事件當入口，但私訊沒有客人後續），這類不是真流失，是資料口徑造成的，解讀時要分開看。

## 二、為什麼會流失：終點對應到對話的哪一步

把終點和對話流程對齊，可以看到流失集中在「漏斗最前段」與「轉換臨門一腳」兩處：

```
客人進來 → 點快速回覆 → 客服開場/釐清 → 療程/問價說明 → 要求聯絡/提供預約 → 留聯絡/預約
   │            │              │                │                  │              │
   │       2,091 斷這  2,067 斷這         （承接中）      680+818 斷這      1,365 成功
   │       (24.21%)   (23.93%)                          (16% 臨門流失)    (15.80%)
```

- **最前段流失最大**：快速回覆後無後續（24.21%）＋ 開場釐清後無回覆（23.93%），合計約 **48%**，幾乎一半的對話在「還沒進入實質諮詢」就斷了。
- **臨門一腳也在漏**：要求聯絡方式後無回覆（7.87%）＋ 提供預約/轉私域後無回覆（9.47%），合計約 **17%**，這些客人已經被推到轉換點，卻沒跨過去。

這兩個結論直接決定了智能客服「該優先補什麼」。

## 三、智能客服目前「能接住」什麼（已實作 + 為什麼）

以下都是 Facebook_CS 系統**已經實作**的模組與機制（來源：系統架構知識庫、優化報告 Phase 進度），對應它能覆蓋的終點情境。

### 3.1 已實作的核心模組

| 模組 / 機制 | 技術實作 | 能接住的情境 | 對應終點 |
| --- | --- | --- | --- |
| 入口接收與去重 | FastAPI Webhook、HMAC 驗簽、Redis 去重、Echo 偵測 | 廣告回覆 / 私訊 / 留言進線 | 全部入口識別 |
| 智能路由 | `FacebookAgent` 依情感+意圖分流 | 一般諮詢 vs 負面情緒 vs 隱私 vs 客訴 | 療程探索、問價 |
| RAG 知識回答 | ChromaDB + BM25 + BAAI Reranker + OpenRouter | 療程介紹、恢復期、價格範圍 | 療程/困擾探索、純價格評估 |
| **即時智能追問** | 回答後用 `\|\|\|` 拆兩則，延遲 1.5s 補一句業務關懷 | 回答後主動再問一句，降低「答完就斷」 | **快速回覆後無後續（部分）** |
| 新舊客問候 | 無 `user_session` 才加問候；舊客直接答 | 開場不重複、不機械 | 開場或釐清後無回覆（部分） |
| Lazy profile 蒐集 | 對話中順勢收姓名/年齡/病史，不用表單 | 降低留資料的摩擦 | 要求聯絡方式後無回覆（部分） |
| 情緒安撫循環 | `anger_score>7` 進安撫；3 次失敗轉真人 | 負面情緒、客訴 | 婉拒/客訴 |
| 安全 / 隱私守門 | Prompt injection 過濾、隱私查詢不直答、術後客訴分流轉真人 | 術後異常、賠償、個資查詢 | 高風險 handoff |
| 真人介入暫停 | admin echo → `pause_bot 45min`，不搶話 | 真人接手時 AI 自動讓位 | handoff |

### 3.2 具體範例（已實作行為）

**範例 A — 即時智能追問（已上線機制）**
```
客人：想了解皮秒雷射
AI：皮秒雷射主要針對毛孔粗大與斑點…效果顯著。|||請問您之前有做過雷射相關保養嗎？😊
```
→ 程式把 `|||` 拆成兩則、延遲 1.5 秒發第二則，用一句關懷把球再丟回給客人。**這正是針對「快速回覆後無後續」設計的，但只在同一輪對話內生效。**

**範例 B — 問價回答（RAG）**
```
客人：肉毒多少錢
AI：肉毒價格依部位而定…建議現場評估。|||方便傳張照片讓我幫您初步評估嗎？
```

**範例 C — 隱私守門（已實作）**
```
客人：幫我查我的預約紀錄
AI：為保護您的隱私，無法直接查詢，請留「姓名+電話」請專人為您確認唷！👩‍⚕️
```

### 3.3 測試結果：能跑，但品質尚未過關（事實陳述）

DeepEval 2026-05-26 scored baseline（25 cases）與 Phase 4 最新進度：

| 能力 | 測試分數 | 判讀 |
| --- | --- | --- |
| RAG 不亂編（Faithfulness） | 0.944 → Phase 4C **1.000** | ✅ 回答有根據、不捏造 |
| 答題相關性（Answer Relevancy） | 0.983 | ✅ 答得到點上 |
| 檢索精準度（Retrieval Relevancy） | 0.771 → 0.743 | ⚠️ 知識庫 evidence 不夠，價格類尤其弱 |
| 預約流程完整度（Completeness） | 0.59 → Phase 4 **0.717** | ❌ 未達 0.80 門檻 |
| 多輪上下文 | 0.66 → Phase 4B **0.84** | ⚠️ 接近通過，仍未穩 |
| 繁體中文品質 | **25/25 全 fail**（0.648） | ❌ 內部 `[SPLIT:*]` marker 漏進輸出 |

**重點：系統「跑得動、答得出、不亂編」已經做到；但「預約收尾、多輪記憶、輸出乾淨度」這三項還沒過 gate，整體 gate 目前 = failed。**

## 四、智能客服目前「接不住」什麼（缺口 + 量化）

把終點分析的最大流失，對照系統現況，缺口非常清楚：

| 最大流失終點 | 量化 | 系統現況 | 缺口性質 |
| --- | ---: | --- | --- |
| 快速回覆後無後續 | 2,091（24.21%） | 只有「同一輪」即時追問（`\|\|\|`） | **缺跨時間的再追蹤**：客人這一輪不回，就沒有第二次機會 |
| 開場/釐清後無回覆 | 2,067（23.93%） | 有問候策略，但開場後不回就斷 | 缺「開場後 N 小時沒回 → 換個角度再問」 |
| 提供預約/轉私域後無回覆 | 818（9.47%） | 有 booking flow，但 completeness 0.717 未達標 | 預約流程收尾不穩 + 缺跨日提醒 |
| 要求聯絡方式後無回覆 | 680（7.87%） | 有 lazy profile | 缺「不想留 LINE 時的替代路徑」 |

**核心缺口一句話：目前所有「主動」都發生在『同一通對話、同一個當下』。一旦客人已讀不回、隔天才想起來，系統沒有任何機制把他撈回來。而這正是 80.28% 流失的主體。**

## 五、最大機會：把「無回覆流失」變成「跨時間主動再行銷」

這是投報率最高的補強方向。以下為**建議方案**（目前系統尚未實作，屬待開發），並標明量化潛力與必要的平台限制。

### 5.1 機制設計：分節點、分時間的主動追蹤

不要用同一句「請問還有什麼問題嗎」打天下，依「斷在哪個節點」發不同訊息：

| 斷點（終點分類） | 量化 | 建議追蹤時機 | 建議訊息方向（依現有話術風格） |
| --- | ---: | --- | --- |
| 快速回覆後無後續 | 2,091 | 隔 1 天 | 「您前幾天看的 ___ 療程，想先了解費用、恢復期，還是適合哪種方式？」 |
| 開場/釐清後無回覆 | 2,067 | 隔 1 天 | 直接接續入口情境：「想幫您看 ___ 該怎麼改善，方便說一下最在意的部位嗎？」 |
| 提供預約/轉私域後無回覆 | 818 | 隔 1–2 天 | 「要不要先用線上照片幫您初步評估？不用馬上到診。」 |
| 要求聯絡方式後無回覆 | 680 | 隔 1 天 | 提供替代路徑：「不方便留 LINE 也沒關係，先跟我說想了解哪個部位，我先幫您整理。」 |
| 報價後無回覆 | （廣告回覆 32.89% 為節點流失率） | 隔 1–2 天 | 「剛剛的報價想先線上照片評估，還是直接約門診？」 |

### 5.2 量化潛力（事實基礎 + 明確假設）

- **可觸及母體**：光是前兩大斷點（快速回覆後無後續 + 開場釐清後無回覆）就有 **4,158 段（48.14%）**。
- 目前明確轉換承接為 1,365 段（15.80%）。
- 假設跨日主動追蹤能把「無回覆流失」救回 **5%**（保守，需 A/B 驗證），以 6,934 段流失計算 ≈ **347 段** 被重新喚回，相當於把整體轉換量再墊高約 **+25%**（347 / 1,365）。
- 救回 10% ≈ 693 段，等同轉換量 **約 +50%**。

> 以上 5% / 10% 為情境假設，非實測值；實際成效必須用小流量 A/B 驗證後才能定論。此處只用來說明「為什麼這是投報率最高的方向」。

### 5.3 必要前提：Messenger 24 小時訊息政策（設計時務必確認）

Meta Messenger 平台對「主動發送」有 24 小時視窗限制：超過客人最後互動 24 小時後，一般訊息無法直接送達，需符合特定 message tag、One-Time Notification（OTN/Recurring Notification）或付費訊息等規則。

→ **「隔天主動發訊息」不是改幾行程式就能上線**，必須先確認可用的合規管道（OTN 訂閱、付費 sponsored message，或在 24 小時內完成第一次追蹤）。這是本方案落地的**第一個技術前提**，建議在設計前先釐清。

### 7.4 平行軌：跨日主動追蹤（最大流失補強，呼應第五章）

此功能對應 80.28% 的最大流失、投報率最高，但**有平台前置條件**，建議與測試階段平行推進，不綁在 DeepEval gate 上：

| 階段 | 內容 | 估時 |
| --- | --- | --- |
| 前置 | 確認 Messenger 24h 政策可用管道（OTN / 付費訊息 / 24h 內追蹤） | 與 W1–2 並行 |
| 開發 | 分節點 follow-up 排程器 + 話術（第五章表） | W3–5 |
| 驗證 | 納入 W7–8 Live A/B（快速回覆後 / 報價後 / 要求聯絡後） | W7–8 |

若 24h 合規管道評估卡關，此軌可能順延，但不阻擋主測試階段 M1 / M2。

### 7.5 假設與風險

- **假設**：工程人力連續投入、不做大規模架構重寫（orchestrator 拆分可漸進完成）、價格 KB 內容可即時提供。
- **風險**：
  1. 繁中品質若不只是 marker 洩漏、還涉及語氣與格式門檻，可能多花 3–5 天。
  2. RAG evidence 補齊依賴內容整理速度，是時程最大不確定來源之一。
  3. Live A/B 需足夠進線量才有統計顯著性，驗證時間視流量而定。

```
輸出品質      —                                 ❌ 繁中 25/25 fail
```

**一句話總結：智能客服目前「能正確回答當下問題」已經做到一定水準；但面對佔比 80.28% 的『已讀不回 / 隔天才想起來』流失，系統幾乎是空白的。把『分節點、分時間的跨日主動追蹤』補上（先解決 Messenger 24h 合規前提），是投報率最高的下一步。**
